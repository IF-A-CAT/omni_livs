# Time Synchronization

This document describes the time synchronization strategy used by the device
system.

## Overview

The system uses PTP synchronization for the main networked sensors. LiDAR,
cameras, and the MID360s built-in IMU are synchronized with the host through
PTP, using the hardware clock of the host network interface as the PTP master
clock.

The additional ADIS16465 IMU is synchronized by timestamping its Data Ready
(DR) signal with the host local clock. The DR signal indicates that a new
ADIS16465 sample is ready. The DR signal time differs from the actual sampling
time by about 1 ms.

The GNSS receiver uses its own satellite time reference. Its output time is
converted from UTC or GPS time generated by the receiver into UNIX time before
being used by the system.

## PTP-Based Synchronization

LiDAR, cameras, and the MID360s built-in IMU are synchronized through PTP. In
this setup:

- The host network interface hardware clock acts as the PTP master.
- LiDAR timestamps are aligned to the PTP time domain.
- Camera timing is aligned to the same PTP time domain.
- ICM40609 IMU data associated with the MID360s module follows the same
  synchronized time reference.

The expected PTP synchronization accuracy is in the sub-microsecond to
microsecond range under a properly configured wired network with hardware
timestamping support. The final measured accuracy should be verified on the
actual device using PTP offset logs and sensor timestamp comparison.

## ADIS16465 DR-Based Synchronization

The ADIS16465 does not use the same PTP synchronization path as the networked
sensors. Its synchronization is based on the Data Ready (DR) signal:

- The ADIS16465 outputs a DR signal when a new sample is ready.
- The host records a local timestamp when the DR signal is received.
- The host timestamp is used as the time reference for the corresponding
  ADIS16465 sample.
- The DR signal time and the actual sampling time differ by about 1 ms.

This offset should be considered when fusing ADIS16465 data with LiDAR, camera,
MID360s built-in IMU, and GNSS data.

## Camera Triggering

The Hikvision cameras are synchronized through the camera internal action
trigger mechanism. After the cameras are aligned to the PTP time domain, an
internal action command is used to trigger the cameras together, so image frames
from the camera group can be captured with consistent timing.

## GNSS Time Conversion

The GNSS receiver outputs position data with UTC or GPS time generated by the
receiver. This timestamp is converted to UNIX time before being compared with
host-side sensor timestamps and logs.

When GPS week and seconds within the GPS week are used, they are converted into
the corresponding UNIX timestamp.

## GNSS Latency Note

The host receives time through network time synchronization, while GNSS position
data is produced by the GNSS receiver and then transmitted to the host. Because
of this data path, the GNSS output position time can have millisecond-level
latency relative to the host's real time.

This latency should be considered when fusing GNSS data with LiDAR, camera, and
IMU data.
